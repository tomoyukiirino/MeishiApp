# 名刺管理アプリ プロダクト仕様書 v2.0

## 1. プロダクト哲学

このアプリは名刺管理ツールではない。**人の記憶のためのアプリ**である。

名刺は単なるデータカードではなく、人の顔であり、出会いの証である。既存の名刺アプリはすべて「名刺をデータに変換し、CRMに流す」ことを目的としている。営業のための量×スピードの最適化であり、人脈を「リード」として扱う。

このアプリの思想はその対極にある。

- 一人ひとりを覚えるためのツール
- 人脈を大切にする人のためのアプリ
- データは原則端末内に完結。AI機能利用時のみ、明示的同意のうえ一時的に外部処理
- CRM連携もクラウド同期もない。それが弱みではなく最大の強み

**「あなたの名刺は、あなたのiPhoneだけに。」**

### ターゲットユーザー

- 士業（弁護士、税理士、会計士など）
- 経営者・個人事業主
- 医師・研究者
- 人脈の質を重視し、顧客情報のプライバシーに敏感なすべてのプロフェッショナル

CRMは大げさすぎる。名刺管理サービスに他人の情報を預けるのは相手に失礼。そう感じるすべての人がターゲットである。

### 機能の採用・不採用方針

機能の採用・不採用は3段階で判断する。

**思想上入れない（恒久的に排除）:**
- CRM連携
- SNS連携（LinkedIn自動取得など）
- 名刺共有機能（他人に名刺データを送る）
- ネットワーク分析・営業リード化
- チーム共有・複数ユーザー管理

**v1では入れないが将来余地あり:**
- リマインダー（さりげない再会通知）
- バッチスキャン（大量一括処理）
- デジタル名刺交換機能
- 思い出し検索モード（曖昧語の複合検索）

**原則入れない（ユーザーの声次第で検討）:**
- プッシュ通知全般

## 2. プラットフォーム・技術スタック

### プラットフォーム
- **iOS ネイティブアプリ**（SwiftUI）
- 対象: iOS 17+
- 開発言語: Swift
- UI: SwiftUI
- 開発環境: Xcode, Apple Silicon Mac（M4 Pro）

### データストレージ
- **SwiftData** によるローカルDB
- すべてのデータは端末内に保存
- クラウド同期は一切行わない

### セキュリティ
- **生体認証ロック**: アプリ起動時にFace ID / Touch IDによる認証（設定でON/OFF可能）
- **FileProtection**: すべてのデータファイルに `NSFileProtectionComplete` を設定（端末ロック中はファイルにアクセス不可）

### 主要フレームワーク
- **Apple Vision Framework** — OCR（テキスト認識）、顔検出、矩形検出
- **Photos Framework** — カメラロールへのアクセス、写真メタデータ取得
- **Contacts Framework** — iPhoneの連絡先への書き出し
- **AVFoundation** — カメラ制御
- **LocalAuthentication** — 生体認証

### 外部通信（AI構造化機能利用時のみ）

AI構造化機能は以下の2つの接続方式を提供する。ユーザーが選択。

**方式A: Akkuratサブスク（デフォルト）**
- Akkuratが運用するプロキシサーバー経由でLLM APIに接続
- LLMはAkkuratが選定（Claude Sonnet）
- ユーザーは設定不要。サブスク契約のみ
- データはAkkuratのプロキシサーバーを経由する

**方式B: ユーザー自前APIキー**
- ユーザーが自分のAPIキーを設定画面で入力
- LLMプロバイダーを選択可能:
  - Anthropic Claude
  - OpenAI ChatGPT
  - Google Gemini
  - Perplexity
- データはユーザーのiPhoneからLLMプロバイダーに直接送信（Akkuratのサーバーを一切経由しない）
- サブスク不要
- 「データ主権をユーザーの手に」という思想に最も忠実な方式

**共通:**
- ユーザーが選択したモードに応じて、OCRテキストのみ or 名刺画像を送信
- 無料版（API不使用）ではネットワーク通信ゼロ

**プライバシーの強度:** API不使用 > 自前APIキー > Akkuratサブスク
**利便性:** Akkuratサブスク > 自前APIキー > API不使用

### 国際化（i18n）
- UIの文字列はすべてLocalizable.stringsに外出し
- コード内にハードコードしない
- 日付・数値フォーマットはロケール依存
- レイアウトは固定幅にしない（長い言語でも崩れない設計）
- OCRの言語設定はユーザーが選択可能
- **v1ではUIは日本語のみ。ただし構造は多言語展開可能な設計とする**

## 3. ビジネスモデル

### 課金ポイントの後決め設計

開発段階では全機能を実装し、どの機能を有料にするかは後から決定する。すべての機能にフィーチャーフラグ（Feature Flag）を設け、設定ファイルの変更だけで無料/有料の境界を切り替えられる設計とする。

```
FeatureFlags.swift
- isAIStructuringEnabled: Bool    // Claude API構造化
- isFacePhotoEnabled: Bool        // 顔写真の紐づけ
- isFaceSearchEnabled: Bool       // 同一人物検索・逆引き
- isExportEnabled: Bool           // CSV/vCardエクスポート
- isCloudBackupEnabled: Bool      // クラウドバックアップ
- isAdvancedTaggingEnabled: Bool  // タグ機能
// ...必要に応じて追加
```

**開発・テスト段階**: 全フラグをtrue（全機能有効）
**商品化段階**: 実際の利用パターンを見て、どこに課金の壁を置くか決定

### 課金モデルの候補（後日決定）

**推奨: 買い切り + AI機能のみサブスク**
- 買い切り（1,500〜3,000円）で全ローカル機能を開放
- Claude APIを使うAI構造化のみ月額サブスク（300〜500円）
- ローカルアプリは買い切りとの心理的相性が良い
- 「データを預けない」思想と月額SaaS感は噛み合いにくい
- 顔写真や検索を有料壁の内側に入れると、核価値まで有料に見える

**技術的にはフラグで柔軟に対応可能。最終決定は商品化段階で行う。**

いずれの場合も、Claude APIを使う機能は必ず課金対象とする（APIコストが発生するため）。

## 4. 機能仕様

### 4.1 名刺撮影・取り込み

#### カメラ撮影
- カメラ起動 → 名刺を認識（矩形検出: VNDetectRectanglesRequest）
- 自動クロップ・傾き補正
- 表面撮影後「裏面も撮影しますか？」のプロンプト
- 表裏両面を1つの名刺データとして保存

#### ライブラリから取り込み
- カメラロールから名刺画像を選択
- 同様に矩形検出 → 自動クロップ・補正

### 4.2 OCR・構造化

#### ステップ1: オンデバイスOCR（全ユーザー）
- Apple Vision Framework（VNRecognizeTextRequest）
- デフォルト対応言語: 日本語・英語・中国語（簡体字/繁体字）・韓国語
- 設定画面でOCR言語の追加・優先順位付けが可能（フランス語、ドイツ語、スウェーデン語、スペイン語など、Apple Visionがサポートする全言語）
- 言語の自動判別も併用（VNRecognizeTextRequestのautomaticallyDetectsLanguageプロパティ）
- 国際的なカンファレンスや海外での名刺交換にも対応
- 表面・裏面それぞれにOCRを実行
- 抽出テキストを表示

#### ステップ2a: 手動構造化（無料版）

OCRテキストからの構造化は3層で行い、ユーザーの手間を最小化する:

1. **自動推定**: 電話番号形式（0x-xxxx-xxxx）、メール形式（@含む文字列）、URL形式を自動検出し、対応フィールドに自動入力
2. **ユーザー確認**: 推定結果をまず表示し、ユーザーは誤りだけ修正
3. **不完全でも保存可能**: 必須項目は「氏名 or 会社名のどちらか」と「名刺画像」のみ。住所・Webサイト・部署等は未設定のまま保存可能

1件ごとの作業時間が長いと継続率が急落するため、「とりあえず保存して後から整理」を許容する設計とする。

#### ステップ2b: AI自動構造化（サブスク or 自前APIキー）

AI構造化は、接続方式（Akkuratサブスク or 自前APIキー）とプライバシーモードの組み合わせで動作する。

**接続方式（セクション2「外部通信」参照）:**
- Akkuratサブスク: 設定不要、プロキシ経由
- 自前APIキー: ユーザーがLLMプロバイダー（Claude / ChatGPT / Gemini / Perplexity）とAPIキーを設定

**プライバシーモード（設定画面でトグル切り替え）:**

**プライバシー優先モード（デフォルト）:**
- OCR抽出テキスト（画像ではない）をLLM APIに送信
- 名刺画像は端末から一切出ない
- 精度はOCRの抽出品質に依存。レイアウト情報が欠落するため、複雑なレイアウトの名刺では構造化エラーが起こりうる

**精度優先モード:**
- 名刺画像を直接LLM Vision APIに送信
- レイアウト情報を含めた高精度な構造化が可能
- 設定画面に「名刺画像をAIに送信します」と明示

**共通:**
- プロンプトで構造化JSONを要求
- 返却されたデータをフィールドに自動入力
- ユーザーが確認・修正して保存

### 4.3 名刺登録フロー

以下を1つの連続フローで完結させる:

1. 名刺撮影（表面 → 裏面オプション）
2. 自動クロップ・補正
3. OCR実行
4. 構造化（手動 or AI自動）
5. **重複検出**（後述。ここで検出し、メモ入力前にユーザーに通知）
6. **出会いの記録** — 「どこで会いましたか？」（イベント名を入力 or 過去のイベントから選択）+ メモ
7. 保存

**重複検出（ステップ5）**: 構造化の直後、メモ入力画面に進む前に実行する。氏名・会社名・メールアドレスで既存のPersonデータと照合し、一致候補があれば即座に「〇〇さんは既に登録されています。この名刺を〇〇さんに追加しますか？それとも別の方として新規登録しますか？」と表示する。既存Personに追加する場合、新しい名刺と出会いの記録がその人に紐づく。メモや出会いの記録を入力した後に重複が発覚するのは最悪のUXであるため、早期検出を徹底する。

### 4.4 顔写真の紐づけ

#### 登録フロー（v1: 手動で1枚紐づけ）
1. Person詳細画面で「顔写真を追加」をタップ
2. カメラロールが開く
3. 写真を選択（学会の集合写真など）
4. Apple Vision Framework（VNDetectFaceRectanglesRequest）で写真内の顔を自動検出
5. 各顔に枠を表示
6. 該当する人の顔をタップ
7. 顔部分を拡大表示 → 確認して保存
8. Personデータに紐づけ完了

v1ではここまで。顔写真を1枚手動で紐づけるだけの軽量な機能。CoreMLモデルやEmbeddingは不要。

#### 同一人物検索（v2で実装）
- Personに紐づけた顔写真から、カメラロール内の同一人物が写っている写真を検索
- 顔特徴量（embedding）によるコサイン類似度比較
- 「この方の写真をもっと見る」機能
- 写真の撮影日・位置情報から「いつ・どこで」がわかる
- 名刺登録日より前の写真が見つかれば「以前にも会っていた」ことがわかる

#### 写真から名刺を逆引き（顔→Person検索）（v2で実装）
「名前が思い出せないが、顔は覚えている」「この写真に写っている人、名刺交換したはずだが誰だったか」というケースに対応する。

**フロー:**
1. 「写真から探す」ボタン
2. カメラロールから写真を選択
3. 顔検出 → 各顔に枠 → タップで選択
4. CoreMLモデルでEmbedding抽出
5. 登録済み全PersonのfaceEmbeddingと比較
6. 一致するPersonがあれば詳細画面へ。候補複数なら類似度順にリスト表示

#### 顔Embeddingの技術アーキテクチャ（v2で実装）
Apple Vision Frameworkは顔の検出・位置取得には優れるが、顔同士を比較するための特徴量ベクトル（Embedding）を出力する公開APIを提供していない。そのため、以下の2段構えで実装する:

1. **顔検出**: Vision Framework（VNDetectFaceRectanglesRequest）で顔の位置を検出し、矩形で切り抜き
2. **特徴量抽出**: 切り抜いた顔画像を、CoreML形式に変換した軽量な顔認識モデル（MobileFaceNet等）に入力し、Embedding配列（`[Float]`）を取得

完全オンデバイスで高速。ネットワーク通信不要。CoreMLモデルはアプリバンドルに同梱（数MB程度）。

#### パフォーマンス設計（スマートスコーピング）（v2で実装）
カメラロール全体をスキャンするのではなく、検索範囲を段階的に絞り込む。

**第1段階（高速・高精度）:**
- 出会いの記録日の前後数日間 + 位置情報周辺の写真に絞って顔検索

**第2段階（ユーザー操作時のみ）:**
- 「範囲を広げて検索」ボタンで、1ヶ月 → 3ヶ月 → 1年と段階的に拡大

### 4.5 一覧・検索

#### 表示モード（切り替え可能）
1. **リスト表示**（デフォルト） — 名前・会社名・役職。顔写真がある人はサムネイル表示、ない人はイニシャルアイコン
2. **顔写真グリッド表示** — 顔写真が紐づいた人のみ表示。「人の記憶」モード
3. **名刺画像サムネイル表示** — 名刺画像のサムネイル一覧

#### 検索
- 氏名、会社名、役職、メモ、イベント名、タグで横断検索
- 日本語・英語両方で検索可能（裏面のOCR結果も検索対象）
- インクリメンタルサーチ
- ソート: 最近登録順（デフォルト）、名前順、会社名順

#### タグ機能
- 自由にタグを作成・割り当て（「外科」「学会関係」「税理士さん」など）
- タグで絞り込み表示
- 1人に複数タグ可
- イベント名とは別軸での整理

### 4.6 Person詳細画面

- 顔写真（登録済みの場合）
- 氏名・ふりがな
- 現在の会社名・役職（最新の名刺から）
- メモ
- タグ
- **名刺履歴**: この人のすべての名刺を時系列で表示（転職・異動で名刺が変わった場合）
- **出会いの記録一覧**: いつ・どこで会ったかの履歴
- 「この方の写真をもっと見る」（v2: 顔写真から同一人物検索）
- 「連絡先に保存」ボタン
- 編集・削除

### 4.7 iPhone連絡先への保存

**片方向エクスポートのみ。同期はしない。**

- ワンタップでiPhoneの連絡先アプリに新規追加
- 最新の名刺の構造化データ（氏名、電話、メール、住所、会社名、役職）をContactsフレームワーク経由で保存
- 保存前にプレビュー画面を表示し、ユーザーが確認してから保存
- 既に同名の連絡先がある場合は警告（自動上書き・マージはしない）
- 連絡先側で修正してもアプリには反映されない（逆も同様）
- 名刺アプリを開かなくても電話帳から直接電話・メール可能

### 4.8 編集・削除

- 全フィールドの手動編集
- 名刺画像の再撮影・差し替え
- 顔写真の変更・削除
- メモ・タグの追加・変更
- Person / 名刺 / 出会いの記録の個別削除（確認ダイアログ付き）

### 4.9 Person統合（マージ）

別々のPersonとして登録してしまった同一人物を、後から1つに統合する機能。英語名刺と日本語名刺で名前の表記が異なる場合や、重複検出をすり抜けて2回登録してしまった場合に使用する。

**フロー:**
1. Person詳細画面で「この人を別の人と統合する」をタップ
2. Person一覧が開き、統合先のPersonを選択
3. 確認画面で以下を表示:
   - 「山田太郎 と Taro Yamada を統合します」
   - どちらの名前を表示名にするか選択（デフォルトは新しいほう）
   - 顔写真がある場合はどちらを使うか選択
   - 「名刺と出会いの記録がすべてまとめられます。この操作は取り消せません。」
4. 統合実行:
   - 統合元のBusinessCard, Encounterをすべて統合先のPersonに移動
   - タグは両方のものを合算
   - メモは統合（改行で連結、または統合先を優先しユーザーが編集可能）
   - primaryCompany, primaryTitleは最新の名刺から自動更新
   - 統合元のPersonを削除

**表記の扱い:**
- Personの `name` には、ユーザーが選んだ表示名を保存
- 選ばなかったほうの名前は消えない（名刺のOCRテキスト内に残っている）
- 検索すれば日本語名でも英語名でもヒットする（OCR横断検索で対応）

### 4.9 エクスポート

- **CSV形式** — 全データまたはタグ/イベントで絞り込んだデータ
- **vCard/VCF形式** — 他のアプリやPCへの移行用
- 共有シート経由でAirDrop、メール添付等で書き出し

### 4.10 バックアップ・リストア

#### 自動バックアップ（オプション）
- デフォルト: OFF（iPhoneのシステムバックアップに含まれるため最低限は守られる）
- 設定でバックアップ先を選択して有効化:
  - iCloud Drive（v1で対応。ネイティブFileManager APIで直接アクセス可能）
  - Google Drive（v2以降。OAuth認証 + SDK）
  - Dropbox（v2以降。OAuth認証 + SDK）
  - OneDrive（v2以降。OAuth認証 + SDK）
- 選択したストレージ内に専用フォルダを自動作成
- 定期的に自動バックアップ（差分更新）

#### バックアップファイル形式
このアプリの思想は「データ主権をユーザーの手に」である。アプリ開発が将来終了した場合でも、ユーザーが大切なデータを取り出せなくなってはならない。そのため、バックアップは標準的なオープンフォーマットで出力する。

- **SQLiteファイル** + **画像フォルダ**（JPEG）を **ZIP形式** でアーカイブ
- ZIPの中身: `database.sqlite` + `images/` フォルダ（名刺画像・顔写真をJPEGで格納）
- 汎用フォーマットのため、将来的にPythonやその他のツールでデータを読み取り・移行することが可能
- 独自バイナリ形式によるロックインは行わない

#### 手動バックアップ
- アプリ内から「バックアップを作成」で随時実行可能
- ファイルアプリ（Files）経由でAirDrop、NAS（SMB接続先）等に保存可能

#### リストア
- 自動バックアップ先から復元
- または手動でバックアップファイルを選択して復元
- 端末買い替え時のデータ移行手段

## 5. UI/UXガイドライン

### デザイン方針
- シンプルで上品。ITに詳しくない士業・経営者が迷わず使える
- 名刺という「紙」の質感を大切にした、温かみのあるデザイン
- 過度な装飾を避け、顔写真と名刺が主役になるレイアウト
- ダークモード対応

### 名刺登録後の導線
- スキャン完了後、すぐにメモ入力欄が表示される（記憶が鮮明なうちに書く）
- 出会いの記録もスキャンフローの中で完結（別画面に遷移しない）
- 不完全な状態でも保存可能（後から編集できる）

### 一覧画面
- デフォルトはリスト表示（スカスカにならない）
- 顔写真グリッドは顔写真の登録数が増えてきたら自然に使い始める想定
- 表示モード切り替えはワンタップ

## 6. データモデル（概要）

### 設計思想: Person中心モデル

このアプリの主語は「名刺」ではなく「人」である。データモデルの中心エンティティは **Person** とし、名刺（BusinessCard）は Person に属する証跡の1つとして位置づける。

この設計により:
- 同じ人が転職して名刺が変わっても、Personに新しい名刺が追加されるだけ
- 学会で毎年会う人の出会いの記録が時系列で蓄積される
- 「以前の所属」と「現在の所属」を自然に区別できる
- 重複検出が「同一人物に名刺を追加する」操作になり、UXが直感的になる

### 画像データの保存方針
画像データ本体はiPhoneのローカルファイルシステム（Documentsディレクトリ内の `images/` フォルダ）にJPEGとして保存し、SwiftData側にはファイル名（相対パス）のみを保存する。SQLiteにData型を直接格納しない。

```
Documents/
├── MeishiData.store      # SwiftData（SQLite）
└── images/
    ├── cards/             # 名刺画像
    │   ├── {uuid}_front.jpg
    │   └── {uuid}_back.jpg
    └── faces/             # 顔写真
        └── {uuid}_face.jpg
```

### Person（人）— 中心エンティティ
- id: UUID
- name: String（氏名）
- nameReading: String?（ふりがな）
- primaryCompany: String?（現在の会社名 — 最新の名刺から自動更新）
- primaryTitle: String?（現在の役職 — 最新の名刺から自動更新）
- memo: String?（この人についてのメモ）
- facePhotoPath: String?（顔写真のファイルパス）
- faceEmbedding: [Float]?（顔特徴量、v2の同一人物検索用）
- tags: [Tag]（タグ、多対多）
- businessCards: [BusinessCard]（この人の名刺一覧）
- encounters: [Encounter]（この人との出会いの記録一覧）
- createdAt: Date
- updatedAt: Date

### BusinessCard（名刺）— Personに属する証跡
- id: UUID
- person: Person（所属するPerson）
- frontImagePath: String（名刺表面画像のファイルパス）
- backImagePath: String?（名刺裏面画像、オプション）
- company: String?（この名刺の会社名）
- department: String?（部署）
- title: String?（役職）
- phoneNumbers: [String]（電話番号、複数可）
- emails: [String]（メールアドレス、複数可）
- address: String?（住所）
- website: String?（Webサイト）
- ocrTextFront: String?（表面OCR生テキスト）
- ocrTextBack: String?（裏面OCR生テキスト）
- acquiredAt: Date（この名刺を受け取った日）
- createdAt: Date

### Encounter（出会いの記録）— Personとの出会い
- id: UUID
- person: Person（出会った人）
- eventName: String?（イベント名）
- date: Date?（日時）
- location: String?（場所）
- memo: String?（この出会いでのメモ。「日本酒が好きと言っていた」等）
- businessCard: BusinessCard?（この出会いで受け取った名刺、オプション）
- createdAt: Date

### Tag（タグ）
- id: UUID
- name: String
- persons: [Person]（多対多）

**注:** 旧仕様のEventモデルは廃止。Encounterに統合することで、二重管理を排除する。

## 7. プライバシーポリシー要旨

ユーザーに明示する内容:

1. **すべてのデータは端末内に保存されます。第三者のサーバーへのアップロードは一切行いません。**
2. **AI構造化機能を使わない場合、ネットワーク通信を一切行いません。**
3. **AI構造化機能を利用する場合、以下の2つの接続方式から選択できます。**
   - **Akkuratサブスク方式:** Akkuratのプロキシサーバー経由でLLM APIに接続します。データはAkkuratのサーバーを経由しますが、保存・蓄積はしません
   - **自前APIキー方式:** ユーザーのiPhoneからLLMプロバイダー（Claude / ChatGPT / Gemini / Perplexity）に直接接続します。Akkuratのサーバーを一切経由しません
   - いずれの方式でも、プライバシー優先モード（テキストのみ送信）と精度優先モード（画像送信）を選択できます
   - 各LLMプロバイダーのデータ保持・学習ポリシーは各社の規約に準じます
4. **顔写真がアプリ外に送信されることはありません。**
5. **バックアップは、ユーザー自身が選択したストレージ（iCloud Drive等）に保存されます。** Akkurat社のサーバーを経由することはありません。
6. **アプリは生体認証（Face ID / Touch ID）によるロックに対応しています。**
